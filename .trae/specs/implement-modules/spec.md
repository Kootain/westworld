# Implement Modules Spec

## Why
在完成详细设计之后，我们需要将 `docs/design/` 目录下的设计文档转化为可运行的系统代码。为了确保系统稳定，我们需要严格按照模块依赖关系（core -> io/memory -> agent/box -> graph）分批次进行实现。每批次实现都需要包含单元测试，并且在测试通过后，将实现过程中的反思总结追加到对应的设计文档末尾，以便后续模块参考。

## What Changes
- 实现 `core` 模块：包含枚举、数据模型、配置、工具类，并编写相关单元测试。
- 实现 `io` 模块：包含输入/输出适配器及接口，并编写相关单元测试。
- 实现 `memory` 模块：包含三级记忆的数据结构和存取逻辑，并编写相关单元测试。
- 实现 `agent` 模块：包含认知节点逻辑和 Prompt，并编写相关单元测试。
- 实现 `box` 模块：包含状态守卫、隐私过滤等系统级拦截，并编写相关单元测试。
- 实现 `graph` 模块：包含 LangGraph 状态定义和图执行流的编排，并编写相关单元测试。
- 所有实现的模块都会在测试通过后，对设计和编码过程进行反思，并将反思结果追加到各自对应的 `docs/design/` 文档中。

## Impact
- Affected specs: `docs/design/*.md` 文档将会被追加实现反思部分。
- Affected code: `src/` 目录下将新增六大核心模块代码，`tests/` 目录下将新增对应的单元测试代码。

## ADDED Requirements
### Requirement: 基于依赖关系的模块实现与测试
系统模块需要按批次依次开发，每一个模块都需要经过测试验证。测试通过后需要反思并记录，这能为系统的渐进式开发提供质量保障和知识沉淀。

#### Scenario: Success case
- **WHEN** 开发者完成一个批次（如 core 模块）的编码并运行测试时
- **THEN** 测试全部通过，并且 `docs/design/core-module.md` 文档末尾新增了“实现反思”段落。
